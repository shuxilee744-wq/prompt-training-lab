# Prompt Training Lab PRD

## 1. 产品定位

Prompt Training Lab 是一个帮助用户学习、练习和改写 prompt 的训练型网页应用。

它不是 prompt 模板展示站，也不是纯文档教程。它的核心目标是让用户经过一组结构化练习后，能够把模糊需求改写成清楚、可执行、可评估的 prompt。

一句话定位：

> 一个面向中文用户的 Prompt 写作训练场。

## 2. 目标用户

MVP 优先服务新手到中级办公用户。

典型用户：

- 会使用 AI，但经常只写“帮我分析一下”“写得高级一点”的用户
- 内容、运营、产品、教育、咨询等需要稳定产出文字和分析结果的人
- 想把自己的工作需求整理成可复用 prompt 的用户

暂不优先服务：

- 专业 Agent 开发者
- 复杂编程工作流用户
- 多模型评测或自动化 prompt optimization 用户

## 3. 核心学习目标

用户完成 MVP 学习路径后，应掌握以下能力：

1. 能识别一个弱 prompt 的主要问题
2. 能把空泛词改写成具体操作步骤
3. 能给 prompt 补充角色、任务、框架、输出格式和约束
4. 能根据场景写出一个完整 prompt
5. 能用检查清单自评自己的 prompt

## 4. MVP 范围

MVP 做一个完整闭环：

> 学一条规则 -> 看弱/强例子 -> 做一个小练习 -> 得到反馈 -> 修改一次 -> 进入下一课

MVP 包含：

- 学习路径
- 课程页
- 练习页
- Prompt 诊断页
- 模板库
- 本地进度记录
- 基于规则的 prompt 评分

MVP 暂不包含：

- 登录账号
- 云端同步
- 付费系统
- 社区分享
- 多人协作
- 真实 AI API 智能评分
- 大规模课程后台

## 5. 页面结构

### 5.1 首页 / 训练入口

目标：让用户立即开始学习，而不是阅读营销内容。

内容：

- 当前学习进度
- 继续学习按钮
- 今日推荐练习
- Prompt 诊断入口
- 模板库入口

首屏原则：

- 不做大 Hero 宣传页
- 不讲过多产品卖点
- 直接给学习动作

### 5.2 课程页

目标：解释一个核心概念，并展示弱/强 prompt 对照。

每节课包含：

- 核心规则
- 弱写法示例
- 强写法示例
- 差异解释
- 进入练习按钮

MVP 课程：

1. Prompt 的五个部件
2. 角色不是头衔
3. 世界观决定输出方向
4. 把抽象词改成操作步骤
5. 用问题链让 AI 不跑偏
6. 输出格式决定交付质量
7. 风格约束和禁止项
8. 从弱 prompt 到强 prompt
9. 写一个可复用 prompt 模板
10. 用自己的场景完成一次实战

### 5.3 练习页

目标：让用户在具体任务里练习 prompt 改写。

推荐布局：

- 左侧：规则说明、任务、参考例子
- 中间：用户 prompt 编辑器
- 右侧：评分和反馈

练习类型：

- 找问题：指出弱 prompt 的缺陷
- 二选一：判断哪个 prompt 更好
- 填空补全：补角色、框架、输出或禁止项
- 局部改写：把空泛要求改成可执行表达
- 完整生成：根据场景写完整 prompt
- 对照复盘：比较用户版本和参考版本

### 5.4 Prompt 诊断页

目标：用户贴入自己的 prompt，系统给结构化反馈。

诊断维度：

- 角色清晰度
- 任务明确度
- 框架可执行性
- 输出具体度
- 约束有效性
- 空泛词风险

输出：

- 总分
- 分项评分
- 主要问题
- 最小修改建议
- 可选参考改写

### 5.5 模板库

目标：提供可复制、可改写的 prompt 模板。

分类：

- 写作润色
- 文章分析
- 学习辅导
- 会议总结
- 产品调研
- 商业分析
- 决策分析
- 个人复盘

模板详情包含：

- 适用场景
- 完整模板
- 可替换变量
- 为什么这样写
- 一个使用例子

## 6. 学习内容结构

基础模型：

```text
角色 + 任务 + 框架 + 输出 + 约束
```

扩展模型：

```text
角色：你是谁
世界观：你怎么看问题
任务：你要完成什么
输入：用户会提供什么
框架：你按什么维度思考
流程：你先做什么后做什么
输出：结果长成什么样
风格：用什么语气表达
禁止：避免哪些坏输出
标准：如何判断结果好坏
```

## 7. 评分规则

MVP 先使用规则评分，不接 AI API。

### 7.1 角色清晰度

检测点：

- 是否包含“你是”
- 是否说明能力
- 是否说明立场或工作边界

示例反馈：

```text
你写了角色名称，但没有说明这个角色如何判断问题。建议补一句“你不做流水账复述，而是寻找结构性原因”。
```

### 7.2 任务明确度

检测点：

- 是否有明确动词
- 是否说明输入对象
- 是否说明完成目标

### 7.3 框架可执行性

检测点：

- 是否包含维度、步骤、层级或问题链
- 是否只出现“深入、全面、专业”等空泛词

### 7.4 输出具体度

检测点：

- 是否说明输出格式
- 是否说明结果包含哪些部分
- 是否有可交付物形状

### 7.5 约束有效性

检测点：

- 是否有风格要求
- 是否有禁止项
- 是否说明证据、引用、反例或边界

### 7.6 空泛词风险

重点提示词：

- 深入
- 全面
- 专业
- 高级
- 优化
- 分析一下
- 润色一下

规则：

如果出现空泛词，但没有配套框架或标准，则提示用户改写。

## 8. 交互原则

- 用户始终知道下一步做什么
- 每次练习只训练一个主要能力
- 反馈要具体，不只说“不错”
- 系统优先给最小修改建议，而不是直接替用户重写全部
- 弱/强对照要始终可见
- 用户输入区要足够大，像工作台而不是表单

## 9. 界面风格原则

风格方向现在确定，细节在 MVP 实现中逐步落地。

定位：

> 安静、专业、训练感强的 Prompt 写作工作台。

像：

- SaaS 工具
- 在线编辑器
- 专业学习平台
- 训练工作台

不像：

- 营销官网
- 儿童闯关游戏
- 纯文档站
- 炫技型 AI 工具

视觉关键词：

- 清晰
- 克制
- 专业
- 有训练节奏
- 反馈明确
- 对照强烈

基础视觉方向：

- 背景：浅灰白或近白
- 主色：深青或墨蓝
- 强调色：橙色或绿色
- 字体：系统中文字体
- 组件：小圆角、细边框、清晰分区
- 布局：左侧学习说明，中间编辑器，右侧反馈

设计节奏：

1. 需求阶段：确定风格原则和信息架构
2. MVP 阶段：实现基础视觉系统
3. 可用后：再打磨动效、密度、状态和移动端体验

## 10. 数据模型草案

### 10.1 Lesson

```ts
type Lesson = {
  id: string;
  title: string;
  summary: string;
  principle: string;
  weakExample: string;
  strongExample: string;
  explanation: string;
  exerciseIds: string[];
};
```

### 10.2 Exercise

```ts
type Exercise = {
  id: string;
  lessonId: string;
  type: "diagnose" | "choose" | "fill" | "rewrite" | "compose" | "review";
  title: string;
  prompt: string;
  material?: string;
  starter?: string;
  referenceAnswer?: string;
  rubric: RubricItem[];
};
```

### 10.3 RubricItem

```ts
type RubricItem = {
  key: string;
  label: string;
  maxScore: number;
  hint: string;
};
```

### 10.4 UserProgress

```ts
type UserProgress = {
  completedLessonIds: string[];
  completedExerciseIds: string[];
  lastLessonId?: string;
  lastExerciseId?: string;
};
```

## 11. 第一版成功标准

MVP 完成后，用户应该能：

- 从首页进入第一课
- 看懂一个规则和对应例子
- 完成一次 prompt 改写练习
- 获得结构化评分和具体建议
- 修改自己的 prompt
- 保存本地学习进度
- 在模板库复制一个可用 prompt

产品体验标准：

- 用户不用读说明就知道怎么开始
- 弱/强写法对照一眼可见
- 编辑器和反馈面板构成清晰工作台
- 评分反馈能指出具体改法
- 页面不显得像普通文档站
